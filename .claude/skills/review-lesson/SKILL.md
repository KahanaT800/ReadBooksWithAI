# 勘验课程：$ARGUMENTS

你是一位资深技术审稿人。现在请对 ReadBooksWithAI 项目中一课已完成的教材进行系统化勘验（review），找出事实错误、不一致、代码问题和图表缺陷。

## 参数解析

将 `$ARGUMENTS` 拆分为三个部分：
- `BOOK_ID`：书名目录（如 `c-and-pointers`）
- `MODULE`：模块目录（如 `module-00-c-basics`）
- `LESSON`：课程文件名（如 `lesson-01-program-structure`）

目标文件路径：`books/$BOOK_ID/$MODULE/$LESSON.md`

---

## 前置检查

1. 检查目标文件 `books/$BOOK_ID/$MODULE/$LESSON.md` 是否存在
   - **若不存在**：提示用户先运行 `/write-lesson`，停止
2. 检查 `books/$BOOK_ID/BOOK-STYLE.md` 是否存在
   - **若不存在**：提示用户先运行 `/config-book`，停止

---

## 执行流程

严格按以下顺序执行，不可跳步。

### 第一步：读取上下文

1. `books/$BOOK_ID/$MODULE/$LESSON.md` — 待勘验的课程全文
2. `books/$BOOK_ID/BOOK-STYLE.md` — 写作约束
3. `books/$BOOK_ID/BOOK.md` — 书级元数据（目标读者、深度设定）
4. `books/$BOOK_ID/module-index.md` — 本课在整体中的位置
5. `guides/lesson-writing-guide.md` — 通用写作规范
6. `guides/depth-system.md` — 深度标记规范
7. `guides/terminology-glossary.md` — 术语一致性表
8. 如有前序课程，读取前序课程（检查知识衔接）

### 第二步：定位原书内容

根据课程 YAML frontmatter 中的 `original_chapters` 和索引文件中的定位信息：

1. 读取 `books/$BOOK_ID/refs/` 下的相关索引文件
2. **优先使用 epub 文本**（如有 `refs/epub-extracted/` 目录）：
   - 索引中标注了 `chNN.txt:LXX` 格式的 epub 行号，直接用 Read 工具读取对应行号范围
   - epub 纯文本可精确定位，是最高效的事实核查方式
3. **降级到 PDF**（仅当 epub 文本不可用或内容不清晰时）：
   - 按索引中的书页范围读取，需将书页转为 PDF 页码
   - ⚠️ 每次不超过 20 页 PDF

### 第三步：七维勘验

按以下七个维度逐一检查，每个维度输出发现的问题清单。

---

#### 维度 1：事实准确性

逐概念对照原书内容 + 联网搜索，验证课程中每一个技术陈述是否正确。

检查项：
- 数据类型大小、范围是否正确
- 标准规定（C89/C99/C11）的引用是否准确
- 行为描述（如"未定义行为"、"实现定义"）是否与标准一致
- 历史沿革描述是否准确（如"C99 引入"、"C11 废弃"）
- 数量词与实际列举是否匹配（如说"四种"但只列了三种）
- 用 WebSearch 验证有疑问的技术陈述

输出格式：
```
### 维度 1：事实准确性

| # | 位置（行号） | 问题描述 | 原文 | 应改为 | 依据 |
|---|------------|---------|------|-------|------|
| 1 | L263 | ... | ... | ... | ... |
```

---

#### 维度 2：代码验证

对课程中**每一个**代码示例：

1. 提取代码块，保存为临时 `.c` 文件
2. 用 `BOOK-STYLE.md` 中的标准编译命令编译
3. 运行并捕获输出
4. 将实际输出与课程正文中描述的预期输出对比

检查项：
- 代码是否能通过编译（无 error）
- 编译是否有 warning（`-Wall -Wextra`）
- 运行输出是否与正文描述一致
- ❌/✅ 对比示例中，❌ 示例是否确实有问题、✅ 示例是否确实正确
- 代码风格是否符合 `BOOK-STYLE.md`（缩进、命名、注释语言等）

> **注意**：标注为"❌ 错误示例"的代码预期会有编译 warning 或运行时错误，这不算问题。但要验证它确实产生了课程描述的那个错误。

输出格式：
```
### 维度 2：代码验证

| # | 代码位置 | 编译结果 | 运行输出 | 正文预期 | 是否一致 | 问题 |
|---|---------|---------|---------|---------|---------|------|
| 1 | L89-L105 | ✅ 通过 | ... | ... | ✅ | — |
| 2 | L150-L165 | ⚠️ 1 warning | ... | ... | ❌ | 输出不匹配 |
```

---

#### 维度 3：Mermaid 图表验证

对课程中**每一个** Mermaid 代码块：

1. 检查 Mermaid 语法是否正确（能否正常渲染）
2. 核对图表内容与正文描述是否一致：
   - 流程图：节点和箭头方向是否反映正文描述的逻辑
   - 节点文字标注是否准确
   - 是否遗漏了正文提到的关键步骤或分支
   - 条件分支的标注（是/否、真/假）是否正确
3. 检查图表是否过于复杂或过于简略
4. 用 Mermaid CLI（如可用）或手动走读验证渲染结果

检查项：
- 语法有效性：节点 ID 无冲突、箭头语法正确、子图闭合
- 逻辑准确性：流程/关系是否与正文描述完全匹配
- 标注准确性：节点和边的文字是否正确
- 完整性：正文提到的所有关键路径是否在图中体现

输出格式：
```
### 维度 3：Mermaid 图表验证

| # | 图表位置 | 图表类型 | 语法 | 逻辑准确 | 问题描述 |
|---|---------|---------|------|---------|---------|
| 1 | L55-L78 | flowchart TD | ✅ | ❌ | continue 的跳转目标应指向"更新表达式"而非"条件判断" |
```

---

#### 维度 4：内部一致性

检查课程文件**自身内部**的一致性。

检查项：
- 正文中的数量词与实际列举是否匹配（"三种…"后面确实列了三种）
- 表格数据与正文叙述是否矛盾
- 前文提到后文会解释的内容，后文是否确实解释了
- 交叉引用是否指向正确位置
- YAML frontmatter 中的 `concepts` 列表与正文实际概念是否对应
- 前置知识检查的问题是否确实在前序课程中已覆盖

输出格式：
```
### 维度 4：内部一致性

| # | 位置 | 问题描述 | 详情 |
|---|------|---------|------|
```

---

#### 维度 5：术语一致性

对照 `terminology-glossary.md` 和 `BOOK-STYLE.md` 的术语偏好表。

检查项：
- 同一概念在课程中是否始终使用相同中文译名
- 译名是否与术语表一致
- 新术语首次出现时是否有中英文对照定义
- 是否使用了 `BOOK-STYLE.md` 术语表中标注"不用"的译法

输出格式：
```
### 维度 5：术语一致性

| # | 位置 | 术语 | 课程用词 | 标准用词 | 问题 |
|---|------|------|---------|---------|------|
```

---

#### 维度 6：格式规范

对照 `lesson-writing-guide.md` 和 `BOOK-STYLE.md` 的格式要求。

检查项：
- [ ] 每个概念集齐"是什么 + 为什么 + 代码/示例 + 易错点"四段
- [ ] 代码符合 `BOOK-STYLE.md` 完整性要求（头文件、main 函数、编译命令）
- [ ] 每个概念至少一组 ❌/✅ 正反对比
- [ ] 至少一张图（Mermaid 或 ASCII）
- [ ] ⭐ 标记符合 `BOOK-STYLE.md` 触发条件
- [ ] [BEGINNER] 标记符合 `BOOK-STYLE.md` 补充场景
- [ ] 陷阱清单 ≥ 3 条，表格格式
- [ ] 自测题 2-3 道
- [ ] 总行数 ≥ 300
- [ ] 代码行宽 ≤ 80 字符

输出格式：
```
### 维度 6：格式规范

| # | 检查项 | 结果 | 问题描述 |
|---|--------|------|---------|
| 1 | 四段式完整性 | ✅ | — |
| 2 | 代码完整性 | ❌ | 第3个示例缺少 #include |
```

---

#### 维度 7：前后衔接

检查与前序/后序课程的知识边界。

检查项：
- 本课是否使用了前序课程未教的概念（超纲）
- 本课定义的概念是否与前序课程重复
- 本课是否为后续课程留好了接口（在概念串联中提及）
- 前置知识检查的引用路径是否正确

输出格式：
```
### 维度 7：前后衔接

| # | 问题描述 | 涉及课程 | 详情 |
|---|---------|---------|------|
```

---

### 第四步：生成勘验报告

将所有维度的结果汇总为一份完整报告，写入 `books/$BOOK_ID/$MODULE/$LESSON-review.md`。

报告结构：

```markdown
# 勘验报告：[课程标题]

> 勘验时间：[日期]
> 勘验对象：`$MODULE/$LESSON.md`

## 总览

| 维度 | 问题数 | 严重程度 |
|------|--------|---------|
| 1. 事实准确性 | N | 🔴/🟡/🟢 |
| 2. 代码验证 | N | 🔴/🟡/🟢 |
| 3. Mermaid 图表 | N | 🔴/🟡/🟢 |
| 4. 内部一致性 | N | 🔴/🟡/🟢 |
| 5. 术语一致性 | N | 🔴/🟡/🟢 |
| 6. 格式规范 | N | 🔴/🟡/🟢 |
| 7. 前后衔接 | N | 🔴/🟡/🟢 |

严重程度：🔴 必须修复（事实错误/代码不能运行） 🟡 建议修复 🟢 无问题

## 详细发现

[各维度的详细表格]

## 修复建议

[按优先级排序的修复建议列表，每条标注影响范围]
```

### 第五步：展示报告并提出修复

1. 向用户展示报告总览
2. 列出所有 🔴 必须修复的问题
3. 询问用户：
   - "自动修复所有问题" — 直接修改课程文件
   - "逐条确认后修复" — 每个问题单独确认
   - "仅保留报告，手动处理" — 不修改文件

### 第六步：执行修复（如用户选择修复）

1. 按优先级逐条修复：🔴 → 🟡
2. 每次修复使用 Edit 工具精确替换
3. 修复完成后重新检查被修改的部分，确保修复未引入新问题
4. 更新 `terminology-glossary.md`（如术语修正涉及全局变更）

### 第七步：清理

1. 删除勘验报告文件 `$LESSON-review.md`（修复已落地，报告不再需要）
2. 如果课程有实质性修改，更新 `module-index.md` 中的状态为 `👀`（已审阅）

---

## 严重程度判定标准

| 级别 | 标准 | 示例 |
|------|------|------|
| 🔴 必须修复 | 事实错误、代码无法编译/运行、输出不匹配、Mermaid 逻辑错误 | "四种家族"但只列了三种；代码编译报 error；流程图箭头方向反了 |
| 🟡 建议修复 | 描述不精确、风格不一致、缺少标注、术语不统一 | char 的 signed 默认值描述不够精确；缺少 ⭐ 标记 |
| 🟢 无问题 | 检查通过 | — |
