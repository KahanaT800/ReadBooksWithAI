# 配置写作约束：$ARGUMENTS

你是一位教学设计专家。现在请基于用户需求和书籍内容，生成一份针对本书的写作约束文档，让后续 `/write-lesson` 有明确的执行标准。

## 参数

- `BOOK_ID`：从 `$ARGUMENTS` 获取

---

## 前置检查

1. 检查 `books/$BOOK_ID/REQUIREMENTS.md` 是否存在
   - **若不存在**：提示用户先运行 `/init-book $BOOK_ID`，停止
2. 检查 `books/$BOOK_ID/BOOK.md` 是否存在
   - **若不存在**：提示用户先运行 `/plan-book $BOOK_ID`，停止
3. 检查 `books/$BOOK_ID/BOOK-STYLE.md` 是否已存在
   - **若已存在**：展示给用户，询问是"在此基础上调整"还是"重新生成"
   - 若调整：读取现有文件，展示给用户，让用户提出修改意见后更新文件
   - 若重新生成：备份为 `BOOK-STYLE.backup.md`，继续

---

## 执行流程

### 第一步：读取输入

1. `books/$BOOK_ID/REQUIREMENTS.md` — 读者背景、深度设定、开发环境、特别要求
2. `books/$BOOK_ID/BOOK.md` — 学科领域、重组方案、模块概览
3. `books/$BOOK_ID/refs/*-index.md` — 浏览索引了解书的内容范围和关键概念
4. `guides/lesson-writing-guide.md` — 了解通用规范中哪些地方标注了"从 BOOK.md 读取"
5. `guides/depth-system.md` — 深度标记规范

### 第二步：生成写作约束草案

基于上述输入，生成 `BOOK-STYLE.md` 的草案，包含以下六个部分：

---

#### 一、代码/示例规范

根据书的学科领域和开发环境，定义本书的代码完整性要求。

需要明确：
- **完整性标准**：什么样的代码算"完整可运行"？
  - 编程类：需要哪些头文件/导入？需要入口函数吗？
  - 非编程类：示例的完整性如何定义？
- **编译/运行命令**：标准的构建和运行命令是什么？
- **辅助工具命令**：如调试器、内存检查工具等（如有）
- **代码风格**：缩进、命名约定、注释语言等

示例（C 语言书）：
```markdown
## 一、代码规范

### 完整性要求
每个代码示例必须包含 `#include` 头文件和 `int main(void)` 入口函数，能直接保存为 `.c` 文件编译运行。

### 标准编译命令
gcc -Wall -g example.c -o example
./example

### 辅助工具
valgrind --leak-check=full ./example   # 涉及动态内存时
```

示例（Python 书）：
```markdown
## 一、代码规范

### 完整性要求
每个代码示例必须是完整可运行的 .py 文件，包含必要的 import 和 if __name__ == "__main__" 入口。

### 标准运行命令
python3 example.py
```

#### 二、隐性知识清单

这是本文件最核心的部分。基于目标读者的背景，列出**本书读者可能不知道但教材常常默认你知道的知识点**。

生成方法：
1. 从 REQUIREMENTS.md 读取读者背景和薄弱环节
2. 浏览书籍索引的核心概念列表
3. 对每个核心概念追问：这个概念依赖哪些前置知识？目标读者具备吗？
4. 不具备的就是隐性知识，必须在首次涉及时主动讲清楚

输出格式：按主题分组，每条标注在哪个模块/概念首次涉及时需要覆盖。

示例（C 语言书，读者有嵌入式经验但 Linux 不熟）：
```markdown
## 二、隐性知识清单

### 内存相关
- 栈上分配不需要 free，出作用域自动释放 → module-01 首次涉及时讲
- malloc 返回未初始化内存（calloc 才初始化为 0）→ module-01
- free 后指针变野指针，应置 NULL → module-01

### 语言细节
- 数组名在表达式中退化为指针（sizeof 和 & 例外）→ module-03
- char *s = "hello" 在只读区不可改，char s[] = "hello" 在栈上可改 → module-03
- 函数参数永远是值传递（指针也是——传的是地址副本）→ module-02

### 工具相关
- gcc -Wall 不是全部警告，-Wextra 更多 → module-00
- valgrind 检测不到栈上越界（只检测堆）→ module-01
```

#### 三、图表偏好

根据学科特点，定义本书常用的图表类型和使用场景。

需要明确：
- 本书最常用哪几类图？
- 每类图用 Mermaid 还是 ASCII？
- 有没有书特有的图表约定？

示例（C 语言书）：
```markdown
## 三、图表偏好

| 图表类型 | 使用场景 | 推荐方式 |
|---------|---------|---------|
| 内存布局图 | 讲解栈/堆/全局区时 | ASCII（更精确控制地址标注） |
| 指针关系图 | 讲解指针指向时 | ASCII（箭头 + 地址框） |
| 数据结构图 | 链表/树/队列等 | ASCII |
| 编译流程图 | 预处理→编译→汇编→链接 | Mermaid graph LR |
| 函数调用关系 | 递归、回调 | Mermaid graph TD |
```

#### 四、⭐ 深入内容的具体触发条件

通用规范只说"底层原理用 ⭐"，这里列出本书中具体哪些内容应标记为 ⭐。

示例（C 语言书）：
```markdown
## 四、⭐ 深入触发条件

以下内容出现时必须使用 ⭐ 深入标记：
- glibc 源码实现（如 malloc/free 的 ptmalloc2）
- 汇编层面行为（如栈帧的 push/pop）
- 编译器具体实现细节（如 GCC 的结构体对齐策略）
- 内核系统调用（如 brk/mmap）
- C 标准规范精确引用（如 C99/C11 对未定义行为的规定）
```

## 五、[BEGINNER] 补充场景

#### 六、术语翻译偏好

列出本书中可能有争议的术语翻译，确定标准译法。

示例：
```markdown
## 六、术语翻译偏好

| 英文 | 本书译法 | 不用的译法 | 理由 |
|------|---------|----------|------|
| dangling pointer | 野指针 | 悬空指针/悬挂指针 | 业界最通用 |
| dereference | 解引用 | 间接引用 | 更直观 |
| heap | 堆 | — | 不与数据结构"堆"混淆时直接用 |
| stack frame | 栈帧 | 堆栈帧 | 避免"堆栈"引起堆/栈混淆 |
```
